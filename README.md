Вот ответы на контрольные вопросы по разделам облачных IDE, систем сборки (Make) и GitHub Actions.

### **Вопросы 1-11: Облачные IDE и Makefile**

1.  **Приведите примеры облачных IDE.**
    *   **repl.it (Replit):** Одна из самых популярных, поддерживает множество языков.
    *   **GitHub Codespaces:** Полноценная среда разработки, интегрированная с GitHub.
    *   **GitPod:** Быстро разворачивает рабочее пространство из репозитория.
    *   **CodeSandbox:** Ориентирован на веб-разработку (JavaScript, TypeScript).
    *   **AWS Cloud9:** Мощная IDE от Amazon.

2.  **Опишите процесс подключения репозитория в repl.it.**
    *   На главной странице Replit нужно нажать кнопку **"Create Repl"** (или "+").
    *   В открывшемся окне переключиться на вкладку **"Import from GitHub"**.
    *   Вставить URL-адрес существующего репозитория (например, `https://github.com/username/repo`).
    *   Выбрать язык (если не определился автоматически) и нажать **"Import from GitHub"**.
    *   Replit склонирует репозиторий и создаст рабочую среду.

3.  **Что такое «сборка»?**
    **Сборка (Build)** — это процесс преобразования исходного кода (который пишет человек) в готовое приложение или исполняемый файл. Для компилируемых языков (C, C++, Java) это включает компиляцию и компоновку (линковку). Для интерпретируемых (Python, JS) это может означать минификацию, транспиляцию или просто подготовку файлов к запуску.

4.  **Какие утилиты могут использоваться для сборки?**
    *   **Make:** Классическая утилита, использующая Makefile.
    *   **CMake:** Генерирует файлы для других систем сборки (например, Makefile).
    *   **Ninja:** Ориентирована на высокую скорость сборки.
    *   **Gradle:** Мощная система для Java/Kotlin.
    *   **Webpack:** Сборщик для веб-проектов.

5.  **Для чего служит Makefile?**
    Makefile служит для автоматизации сборки проекта. В нем описываются правила, зависимости и команды, необходимые для компиляции программы. Утилита `make`, читая этот файл, определяет, какие файлы изменились, и пересобирает только то, что нужно, экономя время.

6.  **Что такое цель в Makefile? Приведите пример.**
    **Цель (target)** — это имя задачи или файла, который нужно создать. Обычно записывается в начале строки перед двоеточием.
    *Пример:*
    ```makefile
    main: main.o utils.o
    	gcc -o main main.o utils.o
    ```
    Здесь `main` — это цель (конечный исполняемый файл).

7.  **Что такое связка в Makefile? Приведите пример.**
    Вероятно, имеется в виду **правило (rule)**. Однако если под "связкой" понимать конкретную команду или рецепт, то это строка с действием (обычно с отступом в виде табуляции).
    *Пример (команда для очистки):*
    ```makefile
    clean:
    	rm -f *.o main
    ```

8.  **Что такое зависимость в Makefile? Приведите пример.**
    **Зависимость (prerequisite)** — это файлы или другие цели, которые должны существовать или быть обновлены ДО того, как будет выполнена текущая цель. Записываются после цели через двоеточие.
    *Пример:*
    ```makefile
    main.o: main.c
    	gcc -c main.c
    ```
    Здесь `main.c` является зависимостью для цели `main.o`. Цель будет выполнена, только если файл `main.c` новее, чем существующий `main.o`.

9.  **Что такое правило в Makefile? Приведите пример.**
    **Правило** — это полная инструкция, состоящая из цели, зависимостей и рецепта (команд).
    *Пример:*
    ```makefile
    # Цель: программа
    program: main.c
    	gcc main.c -o program
    ```

10. **Что такое макроопределение в Makefile? Приведите пример.**
    **Макроопределение** (или переменная) — это имя, которому присваивается значение (строка), чтобы использовать его многократно.
    *Пример:*
    ```makefile
    CC = gcc
    CFLAGS = -Wall -std=c99

    program: main.c
    	$(CC) $(CFLAGS) main.c -o program
    ```

11. **Какая связка используется для очистки проекта?**
    Обычно используется цель с именем **`clean`**. Она не создает файл, а удаляет результаты предыдущей сборки.
    *Пример:*
    ```makefile
    clean:
    	rm -f *.o program
    ```

### **Вопросы 12-14: Модель КИС (Код, Исходники, Сборка)**

12. **В чем заключается модель КИС для проектов на языке C?**
    Модель КИС (часто это аббревиатура "Код, Исходники, Сборка") подразумевает четкое разделение файлов в структуре каталогов проекта:
    *   **Код (Code):** Исходные файлы (`.c`).
    *   **Исходники (Includes/Headers):** Заголовочные файлы (`.h`).
    *   **Сборка (Build):** Объектные файлы (`.o`) и готовые исполняемые модули.
    Это позволяет отделить логику программы от интерфейсов и результатов компиляции.

13. **Что такое стандарты кодирования и почему рекомендуется им следовать?**
    **Стандарты кодирования** — это свод правил о том, как писать код: отступы, названия переменных, структура функций, комментарии.
    *Почему важно:*
    *   **Читаемость:** Код легче читать другим разработчикам (и себе в будущем).
    *   **Поддерживаемость:** Проще находить ошибки и вносить изменения.
    *   **Единообразие:** Весь проект выглядит как единое целое, независимо от того, кто писал конкретный кусок кода.

14. **Какой должна быть структура проекта согласно модели КИС и правилам сборки?**
    Типичная структура папок:
    ```
    project/
    │
    ├── src/          # Исходные файлы (.c)
    │   └── main.c
    │
    ├── include/      # Заголовочные файлы (.h)
    │   └── functions.h
    │
    ├── obj/          # Объектные файлы (.o) (временные)
    │
    ├── bin/          # Исполняемые файлы (результат сборки)
    │
    └── Makefile      # Файл для автоматизации сборки
    ```

### **Вопросы 15-20: GitHub Actions**

15. **Что такое Github Actions и для каких задач применяется данный функционал?**
    **GitHub Actions** — это встроенная в GitHub система CI/CD (Continuous Integration / Continuous Deployment). Она позволяет автоматизировать рабочие процессы прямо в репозитории.
    *Задачи:*
    *   Автоматический запуск тестов при пуше кода.
    *   Сборка проекта.
    *   Публикация пакетов (Docker образов, npm-пакетов).
    *   Деплой на сервер.
    *   Автоматическое добавление меток к Issues.

16. **Что такое workflow в GitHub Actions? Приведите пример.**
    **Workflow** — это автоматизированный процесс, который вы определяете в файле YAML. Он хранится в папке `.github/workflows`.
    *Пример (файл `.github/workflows/hello.yml`):*
    ```yaml
    name: Hello World Workflow
    on: [push]
    jobs:
      say_hello:
        runs-on: ubuntu-latest
        steps:
          - run: echo "Hello, world!"
    ```

17. **Что такое event в GitHub Actions? Приведите пример.**
    **Event** — это событие в репозитории, которое запускает выполнение workflow.
    *Примеры:*
    *   `push` (кто-то запушил код).
    *   `pull_request` (создан или обновлен PR).
    *   `schedule` (запуск по расписанию, как cron).
    *   `workflow_dispatch` (ручной запуск).

18. **Что такое job в GitHub Actions? Приведите пример.**
    **Job** (Задача) — это набор шагов (`steps`), которые выполняются на одном виртуальном сервере (раннере). Jobs могут выполняться параллельно или последовательно (если указать `needs`).
    *Пример:*
    ```yaml
    jobs:
      test:   # Название задачи
        runs-on: ubuntu-latest
        steps:
          - run: npm install
          - run: npm test
    ```

19. **Что такое step в GitHub Actions? Приведите пример.**
    **Step** (Шаг) — это отдельная инструкция внутри Job. Это может либо команда терминала (`run`), либо готовое действие (`action`).
    *Пример (команда):*
    ```yaml
    steps:
      - name: Создание папки
        run: mkdir build
    ```

20. **Что такое action в GitHub Actions? Приведите пример.**
    **Action** — это готовый переиспользуемый блок кода для выполнения повторяющихся задач. Действия публикуются в Marketplace или могут быть написаны вручную.
    *Пример (использование готового action для клонирования репозитория):*
    ```yaml
    steps:
      - name: Checkout code
        uses: actions/checkout@v4  # Это и есть action
    ```
